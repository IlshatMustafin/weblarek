Стек: HTML, SCSS, TS, Vite

Структура проекта:
- src/ - исходные файлы проекта
- src/components/ - папка с JS компонентами
- src/components/base/ - папка с базовым кодом
- src/components/Models/ — папка с моделями данных
- src/components/views/ — папка с компонентами интерфейса

Важные файлы:
- index.html - HTML-файл главной страницы
- src/types/index.ts - file с типами
- src/main.ts - точка входа приложения (Перезентёр)
- src/scss/styles.scss - корневой файл стилей
- src/utils/constants.ts - файл с константами
- src/utils/utils.ts - файл с утилитами

## Установка и запуск
Для установки всех зависимостей проекта и запуска локального сервера разработки выполните в терминале следующие команды:

```bash
# Установка зависимостей
npm install

# Запуск сервера разработки (Vite)
npm run dev
```

После запуска сервер выдаст локальную ссылку (обычно `http://localhost:5173`), которую нужно открыть в браузере для тестирования.

# Интернет-магазин «Web-Larёk»
«Web-Larёk» — это интернет-магазин с товарами для веб-разработчиков, где пользователи могут просматривать товары, добавлять их в корзину и оформлять заказы. Сайт предоставляет удобный интерфейс с модальными окнами для просмотра деталей товаров, управления корзиной и выбора способа оплаты, обеспечивая полный цикл покупки с отправкой заказов на сервер.

## Архитектурный паттерн приложения

Код приложения разделен на слои согласно парадигме MVP (Model-View-Presenter), которая обеспечивает четкое разделение ответственности между классами слоев Model и View. Каждый слой несет свой смысл и ответственность:

Model - слой данных, отвечает за хранение и изменение данных.  
View - слой представления, отвечает за отображение данных на странице.  
Presenter - презентер содержит основную логику приложения и отвечает за связь представления и данных.

Взаимодействие между классами обеспечивается использованием событийно-ориентированного подхода. Модели и Представления генерируют события при изменении данных или взаимодействии пользователя с приложением, а Презентер обрабатывает эти события используя методы как Моделей, так и Представлений.

### Базовый код

#### Класс Component
Абстрактный базовый класс для всех компонентов интерфейса. Класс является дженериком и принимает в переменной `T` тип данных, которые могут быть переданы в метод `render` для отображения.

Конструктор:  
`protected constructor(protected readonly container: HTMLElement)` - принимает и сохраняет корневой DOM-элемент компонента. Инстанцирование напрямую запрещено.

Поля класса:  
`container: HTMLElement` - поле для хранения корневого DOM элемента компонента.

Методы класса:  
`render(data?: Partial<T>): HTMLElement` - Главный метод класса. Он принимает данные состояния, автоматически записывает эти данные в свойства экземпляра через `Object.assign` (вызывая соответствующие сеттеры дочерних классов) и возвращает ссылку на DOM-элемент `container`.  
`protected setImage(element: HTMLImageElement, src: string, alt?: string): void` - утилитарный метод для модификации DOM-элементов `<img>` и безопасной установки картинки с альтернативным текстом.

#### Класс Api
Содержит в себе базовую логику отправки запросов к серверу.

Конструктор:  
`constructor(readonly baseUrl: string, options: RequestInit = {})` - инициализирует базовый URL и прописывает дефолтный заголовок `'Content-Type': 'application/json'`.

Поля класса:  
`readonly baseUrl: string` - базовый адрес сервера.  
`protected options: RequestInit` - объект с заголовками, которые будут использованы для запросов.

Методы:  
`get<T extends object>(uri: string): Promise<T>` - выполняет типизированный GET запрос на переданный в параметрах ендпоинт и возвращает промис с объектом, которым ответил сервер.  
`post<T extends object>(uri: string, data: object, method: ApiPostMethods = 'POST'): Promise<T>` - принимает объект с данными для JSON-тела запроса и отправляет их на ендпоинт. По умолчанию выполняется `POST` запрос.  
`protected handleResponse<T>(response: Response): Promise<T>` - защищенный метод, проверяющий ответ сервера на корректность. Возвращает распарсенный JSON-промис или отклоняет его с текстом ошибки.

#### Класс EventEmitter
Брокер событий реализует паттерн "Наблюдатель", позволяющий отправлять события и подписываться на события, происходящие в системе. Класс используется для связи слоя данных и представления. Реализует интерфейс `IEvents`.

Конструктор класса не принимает параметров.

Поля класса:  
`_events: Map<EventName, Set<Subscriber>>` - хранит коллекцию подписок на события. Ключи коллекции - названия событий или регулярное выражение, значения - коллекция функций обработчиков.

Методы класса:  
`on<T extends object>(eventName: EventName, callback: (event: T) => void): void` - подписка на событие, принимает название события (строку или RegExp) и функцию обработчик.  
`off(eventName: EventName, callback: Subscriber): void` - снятие конкретного обработчика с события.  
`emit<T extends object>(eventName: string, data?: T): void` - инициализация события с передачей данных. Поддерживает маску `*` для глобального прослушивания всех событий.  
`onAll(callback: (event: EmitterEvent) => void): void` - подписка на абсолютно все события приложения.  
`offAll(): void` - полный сброс всей коллекции подписок.  
`trigger<T extends object>(eventName: string, context?: Partial<T>): (data: T) => void` - возвращает фабричную функцию-триггер, генерирующую требуемое в параметрах событие при её вызове.

## Данные

### Интерфейсы и типы данных

#### `IProduct`
Описывает товар в каталоге интернет-магазина:
- `id: string` — уникальный идентификатор товара;
- `description: string` — подробное описание товара;
- `image: string` — URL изображения товара;
- `title: string` — название товара;
- `category: string` — категория товара;
- `price: number | null` — цена товара (может быть `null`, если цена не указана).

#### `IBuyer`
Данные покупателя для оформления заказа:
- `payment: TPayment | null` — выбранный способ оплаты;
- `email: string` — электронная почта покупателя;
- `phone: string` — номер телефона покупателя;
- `address: string` — адрес доставки.

#### `TPayment`
Тип, определяющий допустимые способы оплаты:
- `'card'` — оплата банковской картой;
- `'cash'` — оплата наличными при получении.

#### `TFormErrors`
Объект, используемый для хранения ошибок валидации данных покупателя. Строго типизирован на основе ключей `IBuyer` через `Partial<Record<keyof IBuyer, string>>`.

#### `IGetProductsResponse`
Ответ сервера на запрос списка товаров:
- `total: number` — общее количество товаров, доступных на сервере;
- `items: IProduct[]` — массив объектов товаров.

#### `IOrderRequest`
Тип данных запроса на создание заказа. Объединяет данные покупателя `IBuyer` с информацией о составе корзины через пересечение типов:
- `items: string[]` — массив идентификаторов (ID) товаров из корзины;
- `total: number` — итоговая подтвержденная сумма заказа.

#### `IOrderResponse`
Ответ сервера на успешное создание и оплату заказа:
- `id: string` — уникальный идентификатор созданного заказа;
- `total: number` — подтверждённая сервером итоговая сумма списания.

#### `IApi`
Интерфейс, описывающий базовые методы отправки HTTP-запросов:
- `get<T extends object>(uri: string): Promise<T>` — метод для выполнения GET-запросов;
- `post<T extends object>(uri: string, data: object, method?: ApiPostMethods): Promise<T>` — метод для выполнения POST/PUT/DELETE запросов.

### Интерфейсы состояний рендеринга (Слой View)

Данные интерфейсы описывают структуру объектов, передаваемых в методы `render()` компонентов представления для динамического обновления DOM-дерева.

#### `IHeader`
Данные для отображения верхней панели сайта:
- `counter: number` — числовое значение счетчика товаров в корзине.

#### `ICatalog`
Данные для контейнера галереи каталога:
- `catalog: HTMLElement[]` — массив готовых, отрендеренных DOM-элементов карточек товаров.

#### `ICard`
Базовые данные, необходимые для отображения любой карточки товара:
- `id: string` — уникальный идентификатор карточки;
- `title: string` — название товара;
- `price: number | null` — числовая стоимость товара (или `null`).

#### `ICatalogCard` (наследует `ICard`)
Данные карточки товара, отображаемой в сетке каталога на главной странице:
- `image: string` — URL-путь к изображению товара;
- `category: string` — текстовое название категории товара.

#### `IBasketCard` (наследует `ICard`)
Данные элемента товара, отображаемого в виде строки внутри списка корзины:
- `index: number` — порядковый номер товара в списке корзины.

#### `ICardPreview` (наследует `ICard`)
Данные для подробного просмотра товара в модальном окне (Preview):
- `image: string` — URL-путь к изображению товара;
- `category: string` — текстовое название категории товара;
- `description: string` — подробное текстовое описание характеристик товара;
- `buttonText: string` — текст, который должен быть выведен на управляющую кнопку;
- `buttonDisabled: boolean` — флаг блокировки управляющей кнопки.

#### `IModalContent`
Данные для универсального модального окна-оболочки:
- `content: HTMLElement` — готовый DOM-элемент, который будет размещен внутри модального окна.

#### `IBasketView`
Данные для отображения компонента корзины:
- `items: HTMLElement[]` — массив отрисованных строк товаров `CardBasket`;
- `total: number` — суммарная стоимость всех добавленных товаров.

#### `IFormState`
Базовое состояние для управления валидацией и отображением ошибок любой формы в приложении:
- `valid: boolean` — флаг доступности кнопки отправки формы;
- `errors: string[]` — массив текстовых сообщений об ошибках, выводимых под полями.

#### `IOrderForm`
Данные формы первого шага оформления заказа:
- `payment: TPayment | null` — выбранный способ оплаты;
- `address: string` — введенный адрес доставки.

#### `IContactsForm`
Данные формы второго шага оформления заказа:
- `email: string` — введенная электронная почта покупателя;
- `phone: string` — введенный номер телефона покупателя.

#### `ISuccess`
Данные для окна успешного завершения покупки:
- `total: number` — окончательная подтвержденная сумма списания для вывода пользователю.

### Модели данных

#### Класс `CatalogModel`
Отвечает за хранение и управление товарами в каталоге интернет-магазина.

Конструктор:
- `constructor(events: IEvents)` — принимает брокера событий для уведомления системы об изменениях.

Поля класса:
- `private products: IProduct[]` — хранит массив всех товаров каталога.
- `private selectedProduct: IProduct | null` — хранит товар, выбранный пользователем для подробного отображения.
- `protected events: IEvents` — брокер событий для инициализации процессов перерисовки интерфейса.

Методы класса:
- `setProducts(products: IProduct[]): void` — сохранение массива товаров и инициация чистого события `'items:changed'` (без передачи данных) для уведомления презентера о необходимости перерисовать галерею.
- `getProducts(): IProduct[]` — получение массива всех товаров из модели.
- `getProductById(id: string): IProduct | undefined` — поиск и получение одного товара по его уникальному идентификатору.
- `setSelectedProduct(product: IProduct): void` — сохранение товара для подробного просмотра и генерация чистого события `'preview:changed'` (без передачи объекта) для запуска процесса открытия модального окна.
- `getSelectedProduct(): IProduct | null` — получение товара, выбранного в данный момент для подробного отображения.

#### Класс `CartModel`
Отвечает за управление корзиной покупок — хранением и калькуляцией стоимости товаров, выбранных пользователем.

Конструктор:
- `constructor(events: IEvents)` — принимает брокера событий.

Поля класса:
- `private items: IProduct[]` — динамический массив товаров, находящихся в корзине.
- `protected events: IEvents` — брокер событий для реактивного уведомления презентера об изменениях.

Методы класса:
- `getItems(): IProduct[]` — получение массива товаров, которые находятся в корзине.
- `addItem(product: IProduct): void` — добавление выбранного товара в массив корзины и генерация чистого события `'basket:changed'` для обновления UI.
- `removeItem(id: string): void` — удаление товара из массива корзины по его id и генерация чистого события `'basket:changed'` для обновления UI.
- `clear(): void` — полностью очищает корзину и генерирует событие `'basket:changed'` для сброса счетчиков и интерфейса.
- `getTotalPrice(): number` — подсчет и получение суммарной стоимости всех товаров (товары со значением `price: null` приравниваются к `0`).
- `getItemCount(): number` — получение общего количества товаров, находящихся в корзине.
- `hasItem(id: string): boolean` — проверка наличия товара в корзине по его уникальному идентификатору.

#### Класс `BuyerModel`
Отвечает за хранение, очистку и валидацию данных покупателя, указанных на шагах оформления заказа.

Конструктор:
- `constructor(events: IEvents)` — принимает брокера событий.

Поля класса:
- `private payment: TPayment | null` — выбранный способ оплаты (`'card'` или `'cash'`).
- `private email: string` — электронная почта покупателя.
- `private phone: string` — номер телефона покупателя.
- `private address: string` — адрес доставки заказа.
- `protected events: IEvents` — брокер событий для реактивного извещения презентера об изменениях состояния модели.

Методы класса:
- `setData(data: Partial<IBuyer>): void` — частичное или полное сохранение переданных данных в приватных полях модели. Автоматически вызывает метод `validate()` для пересчета ошибок и инициирует событие `'buyer:changed'`.
- `getData(): IBuyer` — получение всех текущих данных покупателя в виде единого объекта `IBuyer`.
- `clear(): void` — сброс и полная очистка данных покупателя (поля приводятся к пустой строке, оплата в `null`). Автоматически инициирует событие `'buyer:changed'` для очистки инпутов на форме.
- `validate(): TFormErrors` — выполняет валидацию всех полей. Возвращает объект с сообщениями об ошибках `TFormErrors` только для невалидных полей. Если поле успешно прошло валидацию, оно отсутствует в результирующем объекте. Метод является чистым и самостоятельно события в брокер не генерирует.

## Слой коммуникации  

### Класс ApiService
Класс ApiService реализует слой коммуникации приложения и отвечает за взаимодействие с API сервера «Веб‑ларёк». Он использует композицию с классом Api для выполнения HTTP‑запросов.

Конструктор:  
`constructor(apiInstance: IApi)` - принимает экземпляр класса, реализующего интерфейс IApi, для выполнения запросов.

Поля класса:
`private apiInstance: IApi` — экземпляр для выполнения запросов.

Методы класса:
`getProducts(): Promise<IGetProductsResponse>` - Выполняет GET‑запрос к `/product` для получения списка всех товаров.
`placeOrder(orderData: IOrderRequest): Promise<IOrderResponse>` - Отправляет POST‑запрос к `/order` с данными заказа. Возвращает подтверждение от сервера с ID заказа и итоговой суммой.

### Класс Header
Класс Header отвечает за отображение верхней панели страницы (хедера), включая кнопку корзины и счетчика количества товаров. Наследует базовый класс `Component<IHeader>`.

Конструктор:

Сеттеры:
- `set counter(value: number)` — устанавливает текстовое значение внутри counterElement, отображая актуальное количество товаров в корзине.

События: 
- При клике на basketButton инициируется событие через брокер: `events.emit('basket:open')`.

### Класс Card
Абстрактный класс, являющийся общим родителем для всех компонентов отображения карточек товара. Наследует `Component<ICard>`.

Конструктор:
- `constructor(container: HTMLElement)` — принимает корневой элемент разметки карточки и находит в нем общие элементы: название (`.card__title`) и цену (`.card__price`) с помощью утилиты `ensureElement`.

Поля класса:
- `protected titleElement: HTMLElement` — элемент для отображения названия товара.
- `protected priceElement: HTMLElement` — элемент для отображения стоимости товара.

Сеттеры и геттеры:
- `set title(value: string)` — устанавливает название товара в textContent.
- `get title(): string` — вспомогательный геттер, возвращает текущее имя для передачи в alt картинки дочерних классов.
- `set price(value: number | null)` — устанавливает цену товара. Если передано значение null, выводит текст «Бесценно», иначе выводит строку вида `[значение] синапсов`.

### Класс CardCatalog
Класс отвечает за отображение карточки товара в каталоге сетки главной страницы. Наследует Card.

Конструктор:

Поля класса:
- `protected imageElement: HTMLImageElement` — изображение товара (`.card__image`).
- `protected categoryElement: HTMLElement` — категория товара (`.card__category`).

Сеттеры:
- `set image(value: string)` — передает ссылку на картинку в метод `setImage`, подставляя `this.title` в атрибут alt.
- `set category(value: string)` — устанавливает категорию товара и динамически назначает цветовой CSS-модификатор на основе константы `categoryMap` (при отсутствии совпадений ставит `card__category_other`).

### Класс CardPreview
Класс отвечает за отображение подробной информации о товаре в модальном окне (превью). Наследует Card. Является полностью пассивным и лишен внутренней логики принятия решений.

Конструктор:
- `constructor(container: HTMLElement, actions?: ICardActions)` — принимает элемент шаблона полного описания карточки и объект действий. Навешивает коллбэк клика из действий на кнопку `actionButton`. Полностью изолирован от глобальных событий брокера.

Поля класса:
- `protected imageElement: HTMLImageElement` — элемент для вывода изображения товара (`.card__image`).
- `protected categoryElement: HTMLElement` — элемент для отображения категории товара (`.card__category`).
- `protected descriptionElement: HTMLElement` — элемент для вывода подробного описания товара (`.card__text`).
- `protected actionButton: HTMLButtonElement` — кнопка для взаимодействия с товаром (`.card__button`).

Сеттеры:

### Класс Modal
Класс является универсальным контейнером для отображения любого динамического контента в модальном окне. Наследует базовый класс `Component<IModalContent>`.

Конструктор:
- `constructor(container: HTMLElement, events: IEvents)` — принимает корневой элемент разметки модального окна (`.modal`) и объект брокера событий. Вешает вызов `close()` на кнопку закрытия (`.modal__close`) и на клик по оверлею (`e.target === this.container`). На внутренний `.modal__container` вешает `e.stopPropagation()`.

Поля класса:
- `protected closeButton: HTMLButtonElement` — элемент кнопки закрытия модального окна.
- `protected contentElement: HTMLElement` — элемент-контейнер (`.modal__content`), внутрь которого вставляется разметка контента.
- `protected events: IEvents` — брокер событий.

Сеттеры:
- `set content(value: HTMLElement)` — принимает любой готовый DOM-элемент и заменяет им текущее содержимое контента через `replaceChildren`.

Методы:
- `open(): void` — добавляет корневому контейнеру класс активности `modal_active` и инициирует событие `events.emit('modal:open')`.
- `close(): void` — удаляет у корневого контейнера класс `modal_active`, принудительно очищает внутреннее содержимое через `innerHTML = ''` и генерирует событие `events.emit('modal:close')`.

### Класс Basket
Класс отвечает за отображение содержимого корзины в модальном окне. Он управляет списком строк добавленных товаров, выводом их суммарной стоимости и блокировкой/разблокировкой кнопки оформления заказа. Наследует базовый класс `Component<IBasketView>`.

Конструктор:
- `constructor(container: HTMLElement, events: IEvents)` — принимает элемент шаблона корзины и брокер событий. Навешивает клик на кнопку `orderButton` для генерации события `events.emit('order:open')`. В конце конструктора принудительно вызывает `this.items = []` для блокировки кнопки оформления при первоначальной инициализации объекта до загрузки данных.

Поля класса:
- `protected basketList: HTMLElement` — элемент-список (`.basket__list`), куда динамически вставляются строки товаров.
- `protected totalPriceElement: HTMLElement` — элемент для вывода итоговой стоимости заказа (`.basket__price`).
- `protected orderButton: HTMLButtonElement` — кнопка перехода к оформлению заказа (`.basket__button`).
- `protected events: IEvents` — брокер событий.

Сеттеры:
- `set items(cards: HTMLElement[])` — принимает массив готовых элементов `CardBasket`. Если товары есть — динамически подставляет их элементы в список и активирует кнопку `orderButton` через `removeAttribute('disabled')`. Если массив пуст — полностью очищает список через `replaceChildren()`, блокирует кнопку через `setAttribute('disabled', 'true')`, а надпись «Корзина пуста» выводится автоматически средствами CSS-стилей верстки проекта.
- `set total(value: number)` — устанавливает значение стоимости товаров в формате `[значение] синапсов` внутри totalPriceElement.

События:
- При клике на кнопку orderButton инициируется событие перехода к оформлению заказа: `events.emit('order:open')`.

### Класс Form
Класс инкапсулирует общую логику всех форм в приложении: управление кнопкой отправки, отображение ошибок валидации и сбор данных из полей ввода. Объединяет базовое состояние `IFormState` и уникальный тип полей `T`. Наследует базовый класс `Component<IFormState & T>`.

Конструктор:
- `constructor(container: HTMLFormElement, events: IEvents)` — принимает элемент формы и объект брокера событий. Вешает слушатель `input` на контейнер формы для генерации динамического события изменения полей с передачей вычисляемого объекта `{[fieldName]: value}`. Также вешает слушатель `submit`, предотвращающий дефолтное поведение страницы и вызывающий событие отправки `[имя_формы]:submit`.

Поля класса:
- `protected submitButton: HTMLButtonElement` — кнопка отправки формы (`button[type="submit"]`).
- `protected errorsElement: HTMLElement` — элемент для вывода текста ошибок валидации (`.form__errors`).
- `protected events: IEvents` — брокер событий.

Сеттеры:
- `set valid(value: boolean)` — управляет доступностью кнопки отправки (`submitButton.disabled = !value`).
- `set errors(value: string[])` — принимает массив строк ошибок и выводит их в `errorsElement`, соединяя через запятую и пробел.

*Примечание: Любые методы самостоятельной очистки (вроде `clear()`) из класса исключены по требованию ревьюера для сохранения пассивности слоя представления.*

## Список событий приложения

### События моделей данных (Model)
- `items:changed` — генерируется `CatalogModel` при получении новых товаров с сервера. Презентер перехватывает сигнал, запрашивает актуальный массив товаров через метод `catalogModel.getProducts()` и рендерит карточки `CardCatalog` в галерею на главной странице.
- `preview:changed` — генерируется `CatalogModel` при изменении выбранного для просмотра товара. Презентер ловит сигнал, легально запрашивает выбранный продукт через `catalogModel.getSelectedProduct()`, формирует для пассивного превью текстовые и логические состояния кнопки и выводит карточку `CardPreview` в модальное окно.
- `basket:changed` — генерируется `CartModel` при любом изменении состава корзины (добавление, удаление или очистка). Презентер обновляет счетчик хедера, маппит актуальные товары из `cartModel.getItems()` в строки `CardBasket` со сквозными индексами (`index + 1`) и рендерит их в `basketView` с пересчитанной итоговой ценой.
- `buyer:changed` — генерируется `BuyerModel` внутри методов обновления или очистки данных. Извещает презентер об изменении состояния покупателя. Презентер запрашивает из модели сырые данные (`getData()`) и ошибки валидации (`validate()`), после чего реактивно синхронизирует значения текстовых полей, выводит ошибки и управляет доступностью кнопок на формах `OrderForm` и `ContactsForm`.

### События компонентов представления (View) и Презентера (UI)
- `card:select` — генерируется Презентером в коллбэке карточки каталога при клике на неё. Передает данные товара Презентеру для сохранения в модель `catalogModel.setSelectedProduct(item)`.
- `preview:toggle-basket` — генерируется Презентером в коллбэке кнопки действия на превью карточки при клике. Полностью изолировано от моделей: Презентер перехватывает сигнал и самостоятельно принимает решение о добавлении или удалении товара из модели корзины на основе метода `cartModel.hasItem()`.
- `basket:remove` — генерируется Презентером в коллбэке строки корзины при клике на кнопку удаления. Передает данные товара Презентеру для исключения из модели `cartModel.removeItem(item.id)`.
- `basket:open` — клик по иконке корзины в хедере. Презентер вызывает метод `.render()` на существующем экземпляре `basketView` без аргументов и помещает корневой элемент в `modal.content`.
- `order:open` — клик по кнопке «Оформить» в корзине. Презентер вызывает метод `.render()` формы первого шага без аргументов для сохранения ранее введенных данных пользователя. Дополнительно инициирует вызов `buyerModel.setData({})` для первичной синхронизации полей.
- `/^order:.*:change$/` и `/^contacts:.*:change$/` — перехват ввода в текстовые инпуты форм по регулярному выражению. Частичные данные в формате `{[fieldName]: value}` прозрачно транслируются в модель `buyerModel.setData(data)`.
- `order:payment-change` — клик по кнопкам оплаты в форме. Идентификатор выбранного способа (`'card'` / `'cash'`) сохраняется в модель через `buyerModel.setData`.
- `order:submit` — успешная отправка формы первого шага. Презентер вызывает пассивный метод `.render()` формы контактов второго шага без передачи аргументов и выводит её на экран в модальном окне.
- `contacts:submit` — отправка формы контактов на втором шаге. Презентер комбинирует данные покупателя из `buyerModel.getData()`, массив строк ID товаров из корзины и итоговую стоимость, отправляя объект на сервер через `apiService.placeOrder`. При успешном ответе:
    - Вызывается метод `.render()` на созданном ранее глобальном экземпляре `successView`.
    - Окно рендерится с подтвержденной суммой ответа сервера и помещается в `modal.content`.
    - Вызываются методы полной очистки моделей: `cartModel.clear()` и `buyerModel.clear()`, что автоматически сбрасывает интерфейс и счетчики в `0`.
- `success:close` — генерируется Презентером в коллбэке кнопки закрытия финального окна успешного заказа при клике. Перехватывается Презентером для вызова метода `modal.close()`.
- `modal:open` — активация модального окна. Презентер вызывает метод `page.render({ locked: true })` для принудительной блокировки скролла страницы через специализированный пассивный компонент `Page` по ТЗ.
- `modal:close` — деактивация модального окна. Презентер вызывает метод `page.render({ locked: false })` для восстановления стандартной прокрутки страницы.
